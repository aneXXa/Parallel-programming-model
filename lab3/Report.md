## Task 1

### Задание к лабораторной работе (Task 1)

- Реализовать параллельное умножение плотной матрицы на вектор (DGEMV, тип `double`) с использованием `**std::thread**`.
- Обеспечить **параллельную инициализацию** матрицы `A` и вектора `b`.
- Провести измерения для размеров **N \times N**: `20000` и `40000` при заданном списке числа потоков (до 40 потоков для каждого N).
- Построить графики **ускорения** и **эффективности**; сформулировать выводы о масштабируемости.
- По двум наборам замеров оценить «оптимальное» число потоков метрикой J(p) (скрипт `plot_speedup_task1.py`).

### Цель

Сравнить последовательную и многопоточную реализации DGEMV, оценить S_p = T_{\mathrm{serial}} / T_{\parallel}(p) и E_p = S_p / p, обсудить ограничения роста S_p при увеличении p.

### Методика

- **Последовательная версия:** `matrix_vector_serial` — двойной цикл по строкам (`task1.cpp`).
- **Параллельная версия:** `matrix_vector_parallel` — разбиение строк между потоками.
- **Инициализация:** `initialize_parallel`.
- **Замер:** несколько прогонов (`repeats`), отброс максимума (`drop_max`), усреднение.
- **Метрики в таблицах замеров:** столбцы `T_serial`, `T_parallel`, `speedup`, `efficiency`.

## Task 2

### Задание к лабораторной работе (Task 2)

- Реализовать **клиент–сервер** на C++ с потоками: очередь задач, несколько клиентов, блокирующий запрос результата по `id`.
- Сравнить комбинации **механизма доставки результата** (`slot` vs `promise`) и **контейнера** ожидающих задач (`unordered_map` vs `std::map`).
- Выполнить замеры через `**task2_benchmark`**, выбрать лучшую конфигурацию для вашей машины.

### Цель

Оценить влияние реализации синхронизации и типа ассоциативного контейнера на среднее время обработки смеси задач при фиксированных `n_per_kind` и `repeats`.

### Методика

- **slot:** `std::mutex` + `std::condition_variable` + `std::optional<T>` на задачу.
- **promise:** `std::promise` / `std::shared_future` для сигнала готовности результата.
- **Контейнер по `id`:** `std::unordered_map` (в среднем O(1)) vs `std::map` (O(\log n)).
- Очередь задач: `std::queue` + условная переменная (без busy-wait).
- Клиенты: отдельные потоки (`sin`, `sqrt`, `pow`), результаты записываются в CSV построчно.
- `**task2_benchmark`:** прогон всех четырёх комбинаций; в выходных данных столбец `seconds_per_run_mean` — меньше лучше.

### Выводы (Task 2)

1. На приведённых замерах **лучшее среднее время** у комбинации `**promise-o`** (promise + `std::map`). Разница с `promise-u` небольшая — на другой нагрузке порядок может измениться; ориентируйтесь на свой прогон `task2_benchmark`.
2. Варианты `**promise-***` в этом наборе замеров быстрее `**slot-***`: меньше ручной работы с ожиданием на стороне клиента при типичной схеме «future готов — get()».
3. Для `**std::map**` vs `**unordered_map**` при малых n поиск по `id` не доминирует; итог определяется кэш-поведением и константами реализации — снова смотрите **свои** цифры.
