# Лабораторная работа 2 — Task 2 (численное интегрирование, OpenMP)

## Цель

Сравнить две параллельные реализации численного интегрирования:

- `integrate_omp`: локальная сумма в потоке + один `atomic` при объединении;
- `integrate_omp_atomic`: `atomic` на каждой итерации цикла.

Параметры: `nsteps = 40 000 000`, потоки `1,2,4,7,8,16,20,40`.

## Вычислительный узел

- CPU: Intel(R) Xeon(R) Gold 6248, 80 логических CPU
- NUMA: 2 ноды
- RAM: ~754 GiB
- OS: Ubuntu 22.04.5 LTS

## Методика

- Интегрируемая функция: `exp(-x*x)` на `[-4, 4]`.
- Базовая последовательная версия: `integrate`.
- Параллельные версии: `integrate_omp`, `integrate_omp_atomic`.
- Замер времени: несколько запусков, отброс максимума, усреднение.
- Метрики из CSV:
  - `S_par = T_serial / T_par`,
  - `S_at = T_serial / T_at`.

## Выводы с доказательствами

1. **Вариант `integrate_omp` (локальная сумма + atomic при объединении) масштабируется лучше.**  
   -`S_par` растет от `7.90e-08` (1 поток) до `2.24e-06` (40 потоков), то есть примерно в `28.4` раза.

2. **Вариант `integrate_omp_atomic` (atomic на каждой итерации) почти не масштабируется и быстро выходит на “плато”.**  
   - `S_at` меняется мало: `7.66e-08` (1 поток) и `1.07e-08` (40 потоков), оставаясь порядка `1e-08`.

3. **Причина — стоимость частой синхронизации.**  
   - в `integrate_omp_atomic` atomic выполняется для каждого шага интегрирования, из‑за чего растет конкуренция потоков за общий `sum`; в `integrate_omp` atomic используется только при финальном сложении локальных сумм потоков.

Итог: для задач типа суммирования правильный паттерн — локальное накопление и одно объединение, а не atomic на каждом шаге.
